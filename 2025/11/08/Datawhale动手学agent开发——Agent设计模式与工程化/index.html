Datawhale动手学agent开发——Agent设计模式与工程化

agent 工程化不是简单的工作流，而是面向一个项目乃至工程，从规划到数据处理，再到应用最后可评估效果，工程化标准起来就是设计模式，而 agent 设计模式可以理解为模板（类似于开发的模板，解决痛点的方案）。

以智能电商客服机器人为例，可以总结出一套对话、RAG 类 agent 的设计模式。由此可以推广到大学迎新小助手开发，先进行意图识别，从学校的向量知识库进行检索（宿舍、食堂、社团），通过大语言模型输出给用户。

要想总结 agent 的设计模式，就先要了解 AI 能做什么，发现 AI 应用的开发与普通应用开发的不同之处，最后多做项目总结自己的 agent 设计模式。

AI 的三大核心能力分别为：Planning，Mining，Tool-use

只是给出 AI 的能力可能有点太抽象了，从昭波老师的 PPT 再偷了几张图，以下是三个能力对应的场景

这三种场景就对应着三种的 agent 设计模式，但从根本上来说，设计模式之间的差异性在于资源与架构中的某些部分的侧重不同，因此要总结 agent 的设计模式就需要从两个方面：资源和维度出发

对于资源，我们可以从六个维度来分析 agent 案例，看看某一类的 agent 在资源方面的共性，进而总结出设计模式。

  1. 模型：模型选用（文本还是多模态） → 模型微调 → 模型评估 → 模型使用
  2. 数据：知识库获取 → 数据调优（数据清洗） → 数据质量评估 → 数据使用
  3. 工具：插件（插件可以形成工作流）/ 工作流 / API / 代码片段
  4. 提示词/上下文：prompt 模板 → prompt 调优 → prompt 评估 → prompt 使用
  5. 场景
  6. 算力：云端 or 本地

接下来是架构的六个维度：开发 ⋅ 部署 ⋅ 评估 ⋅ 安全 ⋅ 观测 ⋅ 治理

部署需要考虑采用什么样的底层模型和应用架构方式

  • 底层模型：公有云 or 私有化本地
  • 应用架构方式：浏览器插件/微信小程序/企业微信/集成现有工具/嵌入现有业务流

评估则需要从多方面考虑

  • 模型的评估
  • 数据集的评估
  • 提示词的评估：阿里的百炼，百度的千帆，火山的方舟
  • 服务的评估

安全同样非常重要，借昭波老师举的一个例子，数字人直播时要防止输出不当言论，也要提防不当提示词输入。

对于本次课的学习，主要是从一种更高、更宏观的角度来看待 agent，能对 agent 做出系统性的认识，agent 开发有时候困难的往往不一定是技术方面，而是更多开发之后随之而来的问题。「工欲善其事必先利其器」，在进行开发前有一个总体的设计和规划是很有必要的。同时也很感谢昭波老师的分享，「授人以鱼不如授人以渔」，他并不是直接给出几个常用的 agent 设计模式，而是将总结的方法论教给了我们，有助于我们应对快速发展的agent技术带来的变化。
