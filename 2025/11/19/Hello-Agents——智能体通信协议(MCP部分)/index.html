总结

先来一个定义：MCP——标准化智能体和外部工具/资源的通信方式。

学了这个部分，我才明白了MCP中的C(Context)的含义，智能体和工具之间可以共享上下文。

以前我们有 FunctionCalling实现了工具调用，拓展了LLM 的能力边界，可问题在于如此多的工具，一个个函数写过去得多累，能不能把别人写的工具给我用呢？那简单，把代码打包发过来就好，可问题在于别人写的工具所对应的LLM不同，无法做到开箱即用，你还得费劲巴拉的改成自己的LLM。那该怎么办呢？

于是MCP来了，青天就有了。就如课程所说”MCP 统一了智能体与外部工具的交互方式“，也可以说，MCP就是标准化的工具调用。

MCP支持五种传输方式，内存传输、Stdio Transport - 标准输入输出传输（本地开发）、Stdio Transport - 社区服务器（npx方式）、Stdio Transport with Args - 带参数的命令传输、HTTP/SSE/StreamableHTTP Transport。前四种MCPTool可以直接用，但最后一种远程传输，比较特殊，得使用MCPClient。

课后题

MCP（Model Context Protocol）是智能体与工具通信的标准协议。基于 10.2 节的内容，请深入思考：

提示：这是一道动手实践题，建议实际操作

  • 在 10.2.3 节的 MCP 服务器实现中，我们定义了list_tools、call_tool等核心方法。请扩展这个实现，添加一个新的 MCP 服务器，提供以下工具：（1）数据库查询工具；（2）数据可视化工具；（3）报表生成工具。要求工具之间能够协作完成复杂的数据分析任务。

答：我不会，我后面补。

  • MCP 协议支持”资源”（Resources）和”提示”（Prompts）两个重要概念，但本章主要聚焦于”工具”（Tools）。请查阅 MCP 官方文档，了解 Resources 和 Prompts 的设计目的，并设计一个应用场景，展示如何利用这三个核心概念构建更强大的智能体系统。

答：Resources的设计目的是向Client暴露服务器(前提是服务器愿意给)上的数据和文本，模型可以访问本地的文件、远程的API、访问数据库记录，突然感觉这有点RAG做知识库的意思，

​ Prompts是给模型定义一种文本模板，用户只要传入他想要的参数值，就能实现不同的操作，可以理解为自然语言的函数，向外部暴露形参和函数名，传入不同的实参，返回不同的结果。

设计的应用场景是学术论文写作助手，Tools能执行对word/latex的打开、编辑、保存操作，Resources可以提供本地上存放的领域相关的学术资料或者访问学术网站的返回结果，Prompts可以定义学术论文的写作风格、以及模板(标题用什么字体、一号标题用什么字体等等)。

  • MCP 使用 JSON-RPC 2.0 作为底层通信协议，通过 stdio 进行进程间通信。请分析：这种设计有什么优势和局限性？如果需要支持远程 MCP 服务器（通过 HTTP/WebSocket 访问），应该如何扩展当前的实现？

答：优势：MCP的思想很契合JSON-RPC，无状态的特性保证了 tools 的独立调用，它的结构固定，没有太多复杂的格式，却已经能满足MCP的要求，可以请求、响应、通知，这种可靠的格式便于MCP开发者解析、使用。进程间的通信大大降低了安全风险，不用担心网络监听。最后一点是，大部分语言都有成熟的 json 库，这是MCP良好生态的前提。

局限性：不适合大吞吐量和二进制格式的数据传输(图片、视频、音频)、不支持多连接、连接时不可切换。

扩展不会。
